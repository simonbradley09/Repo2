03 — RAG Workflow
What Happens When You Send a Message

The entire retrieval pipeline is on-device — only the final assembled prompt touches the Anthropic API.

Fig 2 — Complete RAG pipeline: on-device retrieval feeds a single Anthropic API call
Local

Save User Message

The user's question is persisted to the Room database for conversation history.

Local

BM25 Retrieval

All text chunks are loaded from Room. Each chunk is scored against the query using BM25 keyword math (k1=1.5, b=0.75). The top 5 scoring chunks are selected.

Local

Prompt Assembly

A context block is assembled from the 5 chunks with source PDF name and page number labels. The last 20 messages of history are prepended. The final messages array is constructed.

Anthropic API

Claude API Call

A single POST to /v1/messages with the system prompt ("answer only from context, cite sources") and the assembled request body. Claude reads the pasted chunk text and responds.

Local

Save & Display

Claude's response is saved to Room DB and displayed in the chat UI.

⚠ Implications of This Approach

BM25 vs. Semantic Search: BM25 matches on exact/stem word overlap. If you ask "what are the financial risks?" and the PDF says "fiscal exposure", BM25 scores that chunk low. A vector-embedding approach would catch the paraphrase; this one won't.

Memory Pressure: getAllChunks() loads every chunk from every PDF into memory on every query. For large libraries this gets slow and memory-heavy — there's no pre-filtering by document before scoring.

No Persistent LLM Memory: Claude has no memory between calls. It only knows what's in the current request body — the 5 chunks plus the last 20 messages.

04 — Key Components
Under the Hood

Click any component to see what it does.

Fig 3 — PDF extraction → overlapping chunking → BM25 keyword scoring
🗄

AppDatabase — Room Singleton

▼
Room database with 3 tables: PdfDocumentEntity stores PDF metadata (name, URI, page count). TextChunkEntity stores chunked text linked to a PDF by pdfId, with page number and chunk index. ChatMessageEntity stores full conversation history (role + content).
📄

PdfTextExtractor — PdfBox Android

▼
Uses PdfBox Android to extract text page-by-page via PDFTextStripper. Processes each page independently, preserving page number attribution for downstream citation.
✂️

TextChunker — 400-word Overlapping Chunks

▼
Splits page text into 400-word chunks with 50-word overlap. The overlap prevents answers that straddle a chunk boundary from being missed. Preserves page number attribution for each chunk.
🔍

Bm25Retriever — Pure Kotlin BM25

▼
Pure Kotlin BM25 implementation with k1=1.5 and b=0.75. Scores all stored chunks against the query and returns the top 5. No embeddings, no vector database — purely keyword-based retrieval.
🤖

ClaudeApiClient — Raw HTTP

▼
Raw HttpURLConnection call to https://api.anthropic.com/v1/messages using claude-sonnet-4-6. Max tokens set to 1024. No OkHttp, no Retrofit — minimal dependency footprint. System prompt instructs Claude to answer only from provided context.
🔐

SecurePreferences — AES-256 Encrypted Storage

▼
Uses EncryptedSharedPreferences with AES-256-GCM via AndroidX Security for API key storage. The key never leaves the device in plaintext.

05 — Design Decisions
Notable Trade-offs

Deliberate choices that shape the app's character.

No Embeddings

Retrieval is BM25 only — no vector DB, no semantic search. Simple, fast, and dependency-free, but misses paraphrase matching.

No Backend

Everything runs on-device except the final Claude API call. Zero server infrastructure to maintain.

Encrypted Key Storage

EncryptedSharedPreferences with AES-256-GCM via AndroidX Security. The API key stays safe at rest.

Persistent URI Permissions

takePersistableUriPermission so PDF references survive app restarts without re-asking for access.

Raw HttpURLConnection

No OkHttp, no Retrofit. A single HTTP client with minimal dependency overhead.

No DI Framework

Repositories and clients are instantiated directly in ViewModels. Simple, explicit, no Dagger/Hilt complexity.

06 — Tuning Guide
Highest-Leverage Parameters

Every tuneable value in the codebase and what moving it actually does.

Parameter Current Effect Trade-off
k (top chunks) 5 Answer coverage vs. prompt noise ↑ More coverage, lower relevance per chunk. ↓ Tighter context, risk missing info.
CHUNK_SIZE 400 words Retrieval precision vs. answer completeness ↓ Finer retrieval, may split answers. ↑ More context per chunk, BM25 scores dilute.
OVERLAP 50 words Boundary coverage ↓ Risk boundary gaps. ↑ Diminishing returns, inflates chunk count.
K1 1.5 Term frequency saturation ↓ Binary presence/absence. ↑ Raw frequency dominates, long chunks win.
B 0.75 Length normalization B=1 fully penalizes long docs. B=0 ignores length. Matters with variable page lengths.
max_tokens 1024 Answer length ceiling Too low truncates answers. No cost to raising — you only pay for tokens generated.
history limit 20 messages Conversational coherence vs. context budget ↑ Better follow-ups, eats context window. ↓ Cheaper, degrades multi-turn coherence.
model sonnet-4-6 Quality vs. cost/speed Haiku: faster, cheaper, weaker reasoning. Opus: stronger, more expensive, slower.
system prompt strict grounding Hallucination prevention vs. flexibility "ONLY context" blocks hallucination but refuses general knowledge queries.

⚡ Key Interaction

Raising k while also raising CHUNK_SIZE will quickly crowd out conversation history from the context window, degrading multi-turn coherence even as single-turn retrieval improves. History and retrieved chunks compete for the same context budget — there's no dynamic allocation between them.
